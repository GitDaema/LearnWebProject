# 봉황대학교 AI 웹정보서비스 — 처음부터 끝까지 만들기

이 문서는 "AI 코딩 도우미 없이, 사람이 처음부터 끝까지 이 프로젝트를 만든다면 어떤 순서로, 왜 그렇게 만들어야 하는가"를 설명합니다. Gemini API처럼 프로젝트 자체의 기능으로 필요한 외부 서비스는 그대로 사용한다고 가정합니다. 정확한 목데이터 값이나 픽셀 단위 UI 스펙까지는 다루지 않고, **이 구조와 순서, 그리고 실제로 부딪히게 되는 함정들**에 집중합니다.

---

## 0. 무엇을 만드는가

재학생이 자연어로 질문하면(예: "이번 학기 성적 어때?", "수요일 시간표 보여줘") AI가 의도를 파악해 적절한 화면(카드)으로 안내해주는 대학 종합 웹정보서비스입니다. 동시에, 일반적인 대학 포털처럼 왼쪽 사이드바에서 시간표/성적/출결/등록금/학적/강의계획서/외부링크/개인정보 같은 메뉴를 눌러 **일반 페이지 이동**도 가능해야 합니다. 핵심 설계 원칙은 하나입니다:

> **"AI 채팅"과 "일반 메뉴 탐색"은 서로 다른 두 가지 사용 방식이며, 섞여서는 안 된다.**
> 홈(로고 클릭)에서만 AI에게 자유롭게 물어보고, 사이드바 메뉴는 평범한 웹사이트처럼 정해진 화면으로 즉시 이동한다.

## 1. 기술 스택과 그 이유

| 영역 | 선택 | 이유 |
|---|---|---|
| 프레임워크 | Next.js (App Router) | 파일 기반 라우팅으로 "메뉴 = 실제 경로" 구조를 그대로 구현할 수 있고, API 라우트(`app/api/*/route.ts`)로 백엔드까지 한 프로젝트에서 해결 가능 |
| 언어 | TypeScript | 목데이터 스키마(인터페이스)를 한 번 정의해두면 프론트/백엔드/카드 컴포넌트가 같은 타입을 공유해 불일치를 컴파일 타임에 잡아냄 |
| 스타일 | Tailwind CSS | 카드형 UI를 컴포넌트 단위로 빠르게 반복 제작하기 좋음 |
| 아이콘 | lucide-react | 일관된 아이콘 세트 |
| AI | `@google/generative-ai` (Gemini API) | 자연어 질의 → 의도/파라미터 분류 |

> **중요**: 프레임워크 버전에 따라 관례(파일 구조, `Link`/`params` 시그니처 등)가 실제로 바뀔 수 있습니다. 기억이나 예전 지식에 의존하지 말고, **작업 시작 전에 설치된 버전의 공식 문서(`node_modules/next/dist/docs` 등 로컬 문서 포함)를 먼저 확인**하는 습관을 들이세요. 이 프로젝트에서도 실제로 Next.js 메이저 버전이 올라가며 관례가 달라진 부분이 있었습니다.

## 2. 초기 스캐폴딩

1. `create-next-app` 계열 도구로 TypeScript + Tailwind + App Router 옵션을 켜고 프로젝트를 생성합니다.
2. `package.json`에 `@google/generative-ai`, `lucide-react`를 추가합니다.
3. `.env.local`에 `GEMINI_API_KEY`(서버 전용)를 넣을 자리를 만들어 두되, **키가 없어도 앱이 죽지 않고 "로컬 데모 모드"로 동작하도록** 설계 방향을 처음부터 잡습니다 (아래 5장 참고). 이렇게 해야 API 키 없이도 UI/UX 전체를 시연·개발할 수 있습니다.
4. 루트 레이아웃(`app/layout.tsx`)에서 폰트, 전역 CSS, `<html>/<body>`만 담당하게 하고, 실제 화면 뼈대(사이드바/헤더)는 다음 장에서 별도 레이아웃으로 분리합니다.

## 3. 데이터 모델링 먼저 (`src/data/mockData.ts`)

UI나 API보다 **데이터 모델을 가장 먼저** 설계합니다. 화면과 백엔드가 이 타입들을 공유하기 때문입니다.

핵심 엔티티(대략):
- `StudentProfile` — 이름/학번/학과/학년/지도교수 등
- `TimetableItem` — `subject/professor/room/day/period(number[])`. **`period`를 배열로 설계**해두는 게 핵심입니다 — "3, 4교시 연강" 같은 걸 표현해야 하고, 나중에 "여러 교시를 동시에 질문"하는 기능과도 자연스럽게 맞물립니다.
- `Syllabus`, `CourseRegistration`, `AttendanceItem`, `OfficialLeave`, `SemesterGrade`, `TuitionInvoice`, `RefundRecord`, `ExternalLink`, `AcademicRecord`

그 다음 각 인터페이스에 대한 **모의 데이터**를 채웁니다. 여기서 반드시 지켜야 할 원칙:

> **같은 사실을 여러 군데서 표현하지 마라. 부득이하면 서로 모순되지 않게 유지하라.**

예를 들어 "알고리즘" 과목의 요일/교시는 `timetableData`(시간표), `syllabusData`의 `schedule` 문자열(강의계획서), `courseRegistrations`의 `schedule` 문자열(수강신청) 세 군데에 각각 문자열/구조로 중복 표현되어 있습니다. 나중에 시간표 데이터를 하나 수정했는데 강의계획서 텍스트를 안 고치면, "AI가 서로 다른 말을 한다"는 혼란이 생깁니다(실제로 이 프로젝트에서 발생한 문제입니다). 출결의 `totalHours`처럼 다른 필드에서 파생 가능한 값(주당 교시 수 × 15주)도 마찬가지로 원본이 바뀌면 같이 갱신해야 합니다. 가능하면 이런 파생값은 상수로 계산해서 저장하고, 왜 그 숫자인지 주석으로 남겨두세요.

## 4. UI 컴포넌트: 카드 시스템 (`src/components/cards/*.tsx`)

의도(intent) 하나당 카드 컴포넌트 하나를 만듭니다: `ProfileCard`, `GradeCard`, `TimetableCard`, `AttendanceCard`, `AcademicCard`, `TuitionCard`, `SyllabusCard`, `LinkCard`. 각 카드는:

- `data` prop 하나만 받아 그 안에서 필요한 걸 전부 렌더링 (외부 상태에 의존하지 않음 → 챗봇 답변 아래에도, 전용 메뉴 페이지에도 그대로 재사용 가능)
- 어두운 배경 + `backdrop-blur` + 은은한 색 블러 원 + `rounded-2xl` 테두리로 통일된 "글래스 카드" 톤을 유지 (디자인 시스템은 색상 팔레트 하나만 정하고 카드마다 강조색만 바꾸는 식으로 일관성 유지)

### 4-1. TimetableCard — 배운 함정 하나

주간 시간표를 그리드로 그릴 때, "교시 칸 여러 개를 세로로 합쳐서 하나의 수업 블록"으로 보여줘야 합니다(예: 3~4교시 연강 블록). 이걸 흔히 이렇게 구현합니다: 요일 컬럼을 `flex flex-col`로 만들고, 그 안에서 수업이 시작하는 교시에만 `position: absolute`로 블록을 띄우고 `height`를 계산해서 늘립니다.

**여기서 반드시 기억할 CSS 규칙**: `position: absolute`인 자식은 `flex` 레이아웃에서 완전히 제외됩니다. `top`을 명시하지 않으면 실제 몇 번째 자식이든 상관없이 **항상 컨테이너 맨 위(0)에 붙습니다.** 즉 `height`만 계산하고 `top`을 빼먹으면, 3교시 수업이든 6교시 수업이든 화면에는 전부 "1교시부터 시작하는 것"처럼 보입니다. (이 프로젝트에서 실제로 이 버그가 있었고, 그 결과 사용자가 "데이터가 틀렸다"고 오인하는 연쇄적인 혼란으로 이어졌습니다 — 코드 버그였지만 처음엔 AI 답변 로직 문제로 오인되었습니다.)

올바른 계산:
```
height = 교시수 × (한 칸 높이) + (교시수 - 1) × (칸 사이 간격)
top    = (시작교시 - 1) × (한 칸 높이) + (시작교시 - 1) × (칸 사이 간격)
```
`height`와 `top`을 항상 같은 단위(한 칸 높이 + 간격)로 계산해서 어긋나지 않게 합니다.

### 4-2. 나머지 카드 설계 팁

- 탭이 필요한 카드(성적: 학기별 탭, 등록금: 고지/환불 탭)는 컴포넌트 내부 `useState`로 처리 — 상위 상태 관리 불필요.
- 배열이 올 수도, 단일 객체가 올 수도 있는 카드(성적/링크)는 `Array.isArray(data)`로 정규화해서 하나의 렌더 경로로 합칩니다.

## 5. AI 챗봇 백엔드 (`src/app/api/chat/route.ts`) — 가장 중요한 부분

### 5-1. 기본 골격

1. 클라이언트가 `{ message: string }`을 POST.
2. `GEMINI_API_KEY`가 없으면 즉시 **규칙 기반 폴백**(아래 5-6)으로 응답 — AI 의존 없이도 전체 기능이 데모 가능해야 합니다.
3. 키가 있으면 Gemini에 "이 질문의 의도(intent)와 파라미터(args)와 사용자에게 보여줄 답변 문장(text)을 JSON으로만 반환하라"는 시스템 프롬프트를 보냅니다. 의도는 `get_student_profile`, `get_grades`, `get_timetable`, `get_attendance`, `get_tuition_and_refund`, `get_syllabus`, `request_academic_change`, `get_external_link` 중 하나(또는 잡담이면 `null`).
4. 응답 텍스트에서 JSON을 파싱(마크다운 코드블록 제거 포함), `intent`에 따라 실제 목데이터를 조회해 `data`로 채우고, 그 `data`를 카드 컴포넌트와 함께 프론트로 반환합니다.

### 5-2. 첫 번째 함정 — "AI가 의도만 맞히고 실제 사실은 모른다"

1차 설계에서는 AI에게 의도/파라미터 분류만 시키고, 사용자에게 보여줄 문장(`text`)도 AI가 알아서 자연스럽게 쓰게 했습니다. 문제는 **AI가 실제 데이터를 본 적이 없다**는 것 — "수요일 3교시에 수업 있어?" 같은 구체적 질문에 정확히 답할 수 없고, 얼버무리는 답이 나옵니다.

해결책: **시스템 프롬프트 안에 목데이터베이스 전체를 JSON으로 통째로 넣어서**, 같은 1회 호출 안에서 AI가 그 데이터에 근거해 `text`를 작성하게 합니다. ("의도 분류용 호출 1번 + 데이터 근거 답변용 호출 1번"으로 나누는 방법도 있지만, 아래 5-4에서 설명하듯 무료 API 요청 한도가 매우 빠듯하므로 **호출 횟수를 1회로 유지하는 게 중요**합니다. 다행히 이 프로젝트의 목데이터는 통째로 프롬프트에 넣어도 될 만큼 작습니다.)

### 5-3. 두 번째 함정 — "AI에게 정확한 사실 추출을 맡기면 신뢰할 수 없다"

시간표처럼 **정답이 정확히 하나로 정해진 조회성 질문**은, AI가 자유 텍스트에서 요일/교시를 뽑아내게 하면 "금요일"과 "목요일"을 헷갈리는 등 신뢰도 문제가 생깁니다(실측됨). 원칙:

> **"자연어를 이해하는 것"은 AI에게 맡기되, "정답이 하나로 정해진 사실 조회"는 코드가 직접 원문에서 추출해서 AI의 결과를 덮어쓴다.**

구현: `get_timetable` 의도일 때는 AI가 뭐라고 답했든 무시하고, 원본 사용자 메시지에서 **정규식으로 직접** 요일(`월/화/수/목/금` 또는 `~요일`)과 교시(`\d+\s*교시`)를 뽑아 그 값으로 실제 데이터를 조회하고, 답변 문장도 코드에서 조립합니다. 이렇게 하면 이 카테고리의 질문은 AI 품질과 무관하게 항상 정확합니다.

- **교시는 하나만 뽑지 말고 전역(global) 매칭으로 전부 수집**하세요. "1교시랑 7교시" 같은 복수 질의를 첫 번째 것만 인식하고 나머지를 무시하는 게 실제로 발생한 버그였습니다. 여러 교시를 물었을 때 결과가 전부 같으면(전부 있음/전부 없음) 하나의 예/아니오로 시작해서 나열하고, **결과가 섞여 있으면 억지로 하나의 예/아니오로 뭉개지 말고 교시별 사실만 정확히 나열**하세요.
- 이 패턴은 "정규식으로 뽑을 수 있는 만큼만 결정적으로 처리하고, 그 외의 자유로운 표현(범위, 여러 요일 조합, 상대적 날짜 표현 등)까지 전부 정규식으로 커버하려 들지 마라"는 트레이드오프 위에 서 있습니다. 그 이상으로 자유로운 질문 형식을 지원하고 싶다면, AI SDK가 제공하는 **함수 호출(function calling / tools)** 기능으로 옮겨가는 게 정석입니다 — 자연어 이해는 AI가 스키마에 맞게 구조화하고, 실제 조회/정답 검증은 여전히 코드가 담당하는 구조입니다. 다만 이는 구현 난이도와 API 호출 안정성 검증 부담이 커지므로, **지금 필요한 범위를 벗어나는 유연성까지 미리 준비하지 않는 것**도 중요한 판단입니다.

### 5-4. 세 번째 함정 — 무료 API 요청 한도

무료 티어 Gemini API 키는 요청 한도가 **모델별로 따로, 그리고 매우 낮게** (일부 모델은 하루 20회 수준) 걸려 있습니다. 게다가 "이 모델이 목록에 존재한다"(`ListModels` API)는 것과 "이 키로 실제로 호출할 수 있다"는 것은 다른 문제입니다 — 어떤 모델은 신규 계정에 아예 제공되지 않아 404가 나고, 어떤 모델은 할당량 자체가 0으로 잡혀 있습니다. **문서나 모델 목록만 믿지 말고, 실제로 최소 호출 한 번씩 날려서 검증**해야 합니다.

대응 설계:
1. **모델 폴백 체인**: 시도할 모델 이름을 배열로 두고, 하나가 실패(429/404/5xx)하면 다음 모델로 자동 재시도. 무료 한도가 모델별로 별도 집계되기 때문에 유효합니다. 단, 실제로 호출 가능한 모델만 넣어야 합니다(추측으로 채우면 매번 확정적으로 실패하는 항목이 지연만 유발).
2. **응답 캐시**: 목데이터가 정적이므로 같은 질문의 답은 항상 같습니다. 정규화한 질문 문자열을 키로 짧게(혹은 서버가 떠 있는 동안은 길게) 캐시해서, 반복 질문(추천 질문 버튼 등)이 매번 API를 소모하지 않게 합니다.
3. **모든 모델이 실패하면** 아래 5-6의 규칙 기반 폴백으로 자연스럽게 넘어가서, 최소한 서비스가 500 에러로 죽지는 않게 합니다.

### 5-5. 네 번째 함정 — 한국어 예/아니오 극성

영어와 달리 한국어의 "예/아니오"는 **객관적 사실이 아니라 "질문자가 말한 내용이 맞는지"**를 기준으로 답합니다. "~없지?"처럼 부정으로 물으면 극성이 반대가 됩니다(실제로 없는 게 맞으면 "예, 없습니다"). 이걸 "사실이 참이면 예, 거짓이면 아니오"로 고정해서 구현하면 부정 의문문에서 전부 반대로 답하게 됩니다.

구현 팁: 이런 언어적 뉘앙스는 **경로에 따라 다르게 처리**하는 게 실용적입니다.
- 정답이 정해진 결정적 코드 경로(시간표 존재 여부처럼)는, 거창한 접미사 목록 대신 아주 단순한 휴리스틱 하나로 충분합니다 — 예: 문장에서 "있"과 "없" 중 어느 게 더 뒤에 나오는지 비교. 이걸로 부정/긍정 여부를 판단하고, "사실과 질문 극성이 일치하면 예, 불일치하면 아니오"라는 한 함수로 정리합니다.
- 그 외 자유로운 표현이 오가는 도메인(성적/출결/등록금 등)은 코드로 일일이 부정문을 감지하려 하지 말고, **시스템 프롬프트에 이 언어 규칙 자체를 한 문단으로 설명**해서 AI가 스스로 판단하게 맡깁니다. 규칙을 코드 분기로 나열하는 대신 모델의 언어 이해력에 위임하는 것도 하나의 정당한 설계 선택입니다 — 모든 걸 정규식으로 해결하려 들지 마세요.

### 5-6. 규칙 기반 폴백 (`handleFallbackIntent`)

API 키가 없거나 모든 모델이 실패했을 때 쓰는 최후 수단입니다. 메시지에 포함된 핵심 키워드(성적/시간표/출결/등록금/강의계획서/휴학·복학·자퇴/외부링크/프로필)로 단순 매칭해 의도를 정하고, 시간표 관련 질의는 5-3의 결정적 로직을 그대로 재사용합니다. AI 없이도 모든 카드/기능을 시연할 수 있게 하는 게 목적입니다.

## 6. 라우팅 구조 — "AI 채팅은 홈에서만"

### 6-1. 레이아웃 분리

- 루트 레이아웃(`app/layout.tsx`): `<html>/<body>`, 폰트, 전역 메타데이터만.
- 라우트 그룹 `app/(portal)/layout.tsx` (괄호 폴더는 URL 경로에 영향을 주지 않음): 사이드바 + 상단 헤더 + 모바일 오버레이를 담당하는 클라이언트 컴포넌트. `usePathname()`으로 현재 경로에 맞는 제목/부제목을 헤더에 보여줍니다.

### 6-2. 사이드바는 `Link`, 상태 전달 아님

사이드바 메뉴 항목은 각각 실제 라우트로 연결되는 `next/link`의 `<Link href="...">`입니다. 로고만 `href="/"`로 홈(AI 채팅)으로 이동합니다. **절대 사이드바 클릭이 AI에게 메시지를 보내는 식으로 구현하지 마세요** — 그러면 사용자가 메뉴를 눌렀을 뿐인데 AI 응답을 기다리게 되고, 매번 API 요청 한도를 소모하게 됩니다.

### 6-3. 메뉴 페이지 = 카드 재사용

`app/(portal)/timetable/page.tsx`, `attendance/page.tsx`, `grades/page.tsx`, `tuition/page.tsx`, `links/page.tsx`, `profile/page.tsx`는 목데이터를 직접 import해서 해당 카드 컴포넌트에 그대로 꽂아주는, AI 호출이 전혀 없는 **정적/단순 조회 페이지**입니다. `academic/page.tsx`(휴복학), `syllabus/page.tsx`(강의계획서)처럼 여러 항목 중 하나를 골라야 하는 경우만 페이지 내부에 탭/선택 상태(`useState`)를 둡니다.

### 6-4. 홈 페이지 = AI 채팅

`app/(portal)/page.tsx`만 채팅 상태(`messages`, `input`, `loading`)를 갖고 `/api/chat`을 호출합니다. 환영 메시지, 추천 질문 칩, 메시지 리스트(사용자/AI 버블 + 응답에 딸린 카드 렌더링), 하단 고정 입력창으로 구성합니다.

## 7. 프론트-백엔드 연결

프론트는 `fetch('/api/chat', { method: 'POST', body: JSON.stringify({ message }) })`로 호출하고, 응답의 `intent`값으로 어떤 카드 컴포넌트를 렌더링할지 `switch`로 매핑합니다(카드 컴포넌트 자체가 정한 `data` 모양과 백엔드가 채워주는 `data` 모양이 반드시 일치해야 합니다 — 6장에서 만든 카드 props 타입을 그대로 백엔드 응답 타입과 맞추세요).

`isFallback` 플래그를 응답에 포함시켜, 규칙 기반으로 답한 경우 UI에 작은 배지로 표시해주면(예: "⚡ 규칙 기반 로컬 분석 결과") 사용자가 지금 AI가 답한 건지 아닌지 헷갈리지 않습니다.

## 8. 개발 중 반드시 만나게 되는 함정 모음

이 프로젝트를 실제로 만들면서 반복적으로 부딪혔던, 미리 알아두면 시간을 아낄 수 있는 것들입니다.

- **한글 커맨드라인 인코딩**: Windows Git Bash에서 `curl -d '한글'`처럼 한글을 커맨드라인 인자로 직접 넘기면 인코딩이 깨져서 백엔드가 엉뚱한 값을 받습니다. API를 curl로 테스트할 때는 **UTF-8로 저장한 JSON 파일을 `--data-binary @file.json`으로 넘기는 방식**을 쓰세요. 이 문제를 코드 버그로 착각하고 삽질하기 쉽습니다.
- **개발 서버 좀비 프로세스**: Windows에서 Git Bash의 `pkill`은 `npm run dev`가 실제로 띄운 `node.exe`를 못 죽이는 경우가 흔합니다. 죽었다고 착각한 예전 서버가 계속 포트를 물고 있으면, **최신 코드로 수정했는데도 옛날 응답이 나와서** "고쳤는데 왜 안 고쳐졌지"라는 혼란을 만듭니다. `netstat -ano`로 포트를 점유한 PID를 찾고 `taskkill //PID <pid> //F`로 확실히 종료하세요.
- **AI 응답과 UI 렌더링을 분리해서 의심하기**: "AI가 틀린 답을 한다"는 증상이 실제로는 답변 텍스트는 맞고 **화면(카드)이 데이터를 잘못 그리고 있어서** 생기는 착시일 수 있습니다(4-1의 시간표 그리드 사례). 버그를 백엔드 로직 문제로 단정하기 전에, 채팅이 실제로 출력한 문장 원문과 화면에 보이는 것을 각각 따로 확인하세요.
- **목데이터를 고칠 땐 연쇄 반경을 확인**: 3장에서 언급한 것처럼, 하나의 사실이 여러 필드/문자열에 중복 표현돼 있다면 한 곳만 고치고 끝내지 마세요.
- **모델/버전 문서를 실측으로 확인**: API가 "지원한다"고 나열한 것과 "지금 이 키/계정으로 실제로 동작한다"는 것은 다를 수 있습니다. 가정하지 말고 최소 단위로 직접 호출해서 확인하세요.

## 9. 검증 방법

- `npx tsc --noEmit` — 타입 오류 확인 (카드 props 타입과 백엔드 응답 타입 불일치를 가장 먼저 잡아줍니다)
- `npx next build` — 라우팅 전체가 정상적으로 정적/동적 페이지로 빌드되는지 확인
- API 라우트는 8장의 UTF-8 파일 기반 curl로 개별 시나리오(요일별/교시별/부정문/복수교시/캐시 재사용 등)를 하나씩 검증
- 화면 배치처럼 텍스트로 검증 안 되는 부분(4-1의 그리드 등)은 실제로 브라우저에서 렌더링해서 눈으로 확인
